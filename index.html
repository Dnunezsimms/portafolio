Ciencia de Datos con Rigor Biológico

Mi trabajo se sitúa en la intersección de la agronomía, las ciencias ecológicas y la ingeniería de software. No considero los datos climáticos o ambientales como meras tablas numéricas; entiendo los procesos físicos y fisiológicos subyacentes.

He liderado la consolidación e ingesta de redes climáticas operacionales chilenas (RAN e INIA) y datos satelitales globales (ERA5-Land), aplicando estrictos controles de calidad (QA/QC) y algoritmos de relleno de brechas (gap-filling).

Mi enfoque principal es traducir variables climáticas dinámicas (grados día, porciones de frío, evapotranspiración, déficit de presión de vapor) en modelos predictivos transparentes y explicables que asistan de forma directa decisiones de manejo de recursos e industria.

Engineering Practices

Estándares de desarrollo de software aplicados a la reproducibilidad científica.

Arquitectura de Carpetas Limpia

Los proyectos se estructuran siguiendo pautas profesionales de ingeniería de software para separar el código inmutable del negocio, los scripts de ejecución y las configuraciones:

  • src/: Módulos de Python empaquetados e inmutables (ej. ingestión, procesamiento y lógica de cálculo).
  • config/: Archivos dinámicos de parámetros para evitar variables hardcodeadas.
  • scripts/: Drivers de flujo orquestados secuencialmente (ej. 01_inventory.py, 02_build_sites.py).
  • tests/: Suite de validaciones lógicas para verificar que el código funciona tras modificaciones.

Parámetros Desacoplados

En el proyecto WineWise-RS, toda la asignación de fundos, variables meteorológicas del estudio y directorios locales/remotos se centraliza en archivos de configuración YAML. Esto permite:

  • Portar el pipeline a otros servidores simplemente modificando config/paths.yaml.
  • Ajustar variables de estudio (ej. temperaturas máximas, mínimas o radiación) en config/study_config.yaml sin reescribir la lógica de validación.
  • Evitar brechas de seguridad al separar las llaves de API (guardadas en variables de entorno o archivos .env locales no versionados).

Logging y Trazabilidad en Producción

Para asegurar que las ingestas de datos desde APIs como DataVid sean auditables, se implementa logging formal en los scripts operacionales:

  • Registro detallado de llamadas HTTP, reintentos en fallas de conexión y timeouts de servidores climáticos.
  • Seguimiento de inconsistencias en registros horarios consolidados para auditoría manual.
  • Logs persistentes estructurados en archivos (ej. dashboard_obj3_integrado.log) con marcas de tiempo y niveles de severidad (INFO, WARNING, ERROR).

Pruebas Unitarias para Algoritmos Críticos

La precisión de la acumulación térmica y el mapeo de estaciones depende de algoritmos matemáticos que requieren consistencia lógica. Se utiliza pytest para asegurar la exactitud:

  • Validación de Grilla: Tests para verificar que las coordenadas lat/lon de ERA5-Land se asocian de forma correcta al pixel más cercano del fundo agrícola.
  • Integridad de Fórmulas: Pruebas unitarias sobre el cálculo del biofix de Grados Día (GDD) y porciones de frío invernal.
  • Evita regresiones de software cuando se optimiza el motor de cálculo matemático.

Inmutabilidad de Datos Crudos (QA/QC)

Un principio fundamental del flujo es el respeto a la verdad científica de las observaciones de campo:

  • La carpeta data/raw/ se trata como una base de datos de solo lectura estricta. Ningún script puede alterarla.
  • Todo el proceso de relleno de brechas (gap-filling) y filtrado de sensores defectuosos genera nuevos archivos intermedios en data/processed/.
  • Se documenta la lógica de gap-filling a través de flujogramas explícitos (como gap_filling_decision_flow.png) para mantener transparencia absoluta sobre la manipulación de datos climáticos.
